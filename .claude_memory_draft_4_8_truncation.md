# §4.8 truncation hit 5× in the 2026-05-24 security-health design session

Rémy caught it visually each time — file would render as a broken
half-screen in the browser. Each truncation came from bash heredoc
appends OR from Python script writes through the bash mount on
files ≥100 lines. Pattern:

- `tail` on disk shows last line mid-sentence or empty with NUL bytes
- `check-spec.sh` tail-well-formedness check sometimes catches it,
  sometimes doesn't (passes when NUL bytes pad past a valid-looking
  last line)

**Rule learned: after every bash heredoc write to a non-trivial
file, run a tail-then-html-close verification before reporting
done.** The one-liner that catches both modes:

```bash
for f in <files>; do
  last=$(tr -d '\000' < "$f" | tail -1 | tr -d '[:space:]')
  nuls=$(tr -cd '\000' < "$f" | wc -c)
  [ "$last" = "</html>" ] && [ "$nuls" = "0" ] && echo "OK $f" || echo "FAIL $f"
done
```

For markdown files, swap `</html>` for a known-good terminator
(usually the last `|` of a deferred table row, or a period).
