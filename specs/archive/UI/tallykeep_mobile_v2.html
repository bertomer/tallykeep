TallyKeep — Mobile (v2 draft)

Second mobile pass. Reflects the form-factor decision documented in mobile_form_factor_decision.md: Capacitor wrapper from v1, mobile is a real product (not a companion), spending keys live in Keychain/Keystore. Compared to v1 the header is cleaner, mobile can add Holdings on its own, and Strongbox messaging is softer.
Changes from v1:
  • Header simplified. Unit cycle moved next to the balance amount as a small ⇄ pill. Standalone unit-controls bar removed. Currency picker moved to a tappable secondary line under the balance (and More → Display).
  • Empty state inverted. "Add an Account or Purse from this phone" instead of "go to your desktop". Mobile now owns Holding creation for non-ceremonial types.
  • Strongbox softened. Send still defers to desktop in v1, but message acknowledges v1.x QR-PSBT path is on the way.
  • Settings shows secure storage truth. "Spending keys: stored in iOS Keychain, biometric-protected" — concrete, not vague.
  • New: Add Holding screens. Picker + Purse-add wizard step demonstrate the new mobile-can-add posture.

Promise check — v2
With the form-factor decision locked in (mobile_form_factor_decision.md), the conflict and tension rows from v1's promise check resolve. Here's the updated state.
Locked-in commitment Mobile How / where
No custody, ever — backend never holds Bitcoin keys Held Backend stays watch-only. Spending keys for Purse and Lightning live on the user's phone, in iOS Keychain / Android Keystore — never on TallyKeep infrastructure.
No spending keys in PWA frontend (any platform) Held PWA on desktop or mobile-browser holds zero spending material. Only the Capacitor app holds keys, gated by hardware-backed secure storage.
Banking ergonomics — no crypto-native friction Held Bottom-nav structure. Banking analogies in pickers and tooltips. Header simplified — unit cycle is inline with balance, not a chrome bar. Currency picker is a tap-to-open sheet, not a permanent dropdown.
Sats-default, fiat opt-in Held B1 sats-only default with "Show as fiat →" link. B2 fiat-on with rate attribution. Currency picker on the secondary line itself.
No performance metrics on home Held No daily change indicators anywhere.
Vocabulary preserved (Account, Purse, Strongbox, Vault) Held All four type names canonical in lists, headers, and the Add Holding picker. Banking analogies as descriptions, not replacements.
Declared vs observable security analyser Held B2 surfaces the discrepancy banner inline. Deep "Review" still routes to desktop or full-web for the diagnostic surface.
Lightning keys on-device, not server-side v1.5 Stubbed throughout v1. When shipped: Capacitor only — never PWA. Documented in mobile_form_factor_decision.md.
Mobile is a real product, not a companion Held Account, Purse, Strongbox addable from phone. Sweep policy editable from phone. Send from on-device Purse keys with biometric. Only Vault setup and (v1) Strongbox sends defer to desktop.
Phone-only target market workable Held The LatAm/Africa user can sign in to hosted-tier, add Account + Purse, send and receive — without ever touching a desktop. The §10 amendment makes this real.
v1 backend on localhost, no auth v1 limitation Self-hosted users on the Capacitor app: same-Wi-Fi only in v1, via mDNS or LAN IP at pairing. Real off-network needs v2 (WireGuard/Tailscale). Hosted-tier path bypasses this entirely. Honest in A2 and G1.
Strongbox sends from mobile v1.x Honest stub in v1: "use desktop". v1.x adds QR-PSBT roundtrip for compatible hardware wallets (Coldcard, Jade). Vault stays desktop because the multisig ceremony genuinely doesn't fit a phone.
Open questions
Carried forward from mobile_form_factor_decision.md:
  1. Native secp256k1 plugin in v1 vs v1.5 (does signing happen in native code or in JS after Keychain decrypt).
  2. Reproducible-build target date (v1.x or v1.5).
  3. F-Droid inclusion — depends on Capacitor and Lightning SDK licence chain.
  4. iOS Safari PWA fallback usability — should it support a PSBT-QR roundtrip even though it can't hold keys?
  5. Hosted-tier login UX inside Capacitor app (email + passphrase, OAuth, LNURL-auth?).
  6. ARS as default fiat for Argentine users — given inflation regime, does USD make a saner default? Worth benchmarking against Lemon, Belo, Ripio defaults.
  7. Desktop home page and onboarding HTML (tallykeep_home_final.html, tallykeep_onboarding_validated.html) should adopt the same simplified header pattern (unit cycle inline, currency in secondary line). Not done in this pass.
