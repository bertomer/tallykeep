2 · Philosophy — restraint over decoration

Generic web semantics (saturated green / red / amber, used freely) treat color as decoration — every form passes/fails, every toast cheers or scolds. Banking-grade restraint earns color: green only for confirmed-on-chain with enough depth (per PROCESS.md §3 gauntlet Q4 — confirmation honesty), red only for irreversible-loss risk, amber for action-needed, info muted to near-neutral. The result: most of the UI is text on cream, and color appears where consequence appears.

This locks the mental rule for downstream design. A reviewer should ask, of any proposed use of green: does this represent a verifiably-true positive end state? If not, it shouldn't be green. Same test for red and consequence; same test for amber and pending action.

Hue family — warm-tuned, not generic-modern

The wood palette is yellow-brown, low-mid saturation, mid lightness. A modern saturated palette (Tailwind #16a34a green, #dc2626 red) sits on cream like a foreign body. The locked palette below is warm-tuned — hues drawn from natural references (holly leaf, holly berry, autumn maple, cornflower) — and dialed bright enough to retain signal strength. Specifically not muted-historical (the v0 sketch ran too quiet, especially for danger).

3 · The semantic palette

Each semantic provides four token slots: main (text/icon/border-stripe), soft (banner / badge background tint), border (mid-tone separator on soft-bg components), and text-on-soft (deep variant for legible text inside a soft-bg fill).

4 · Holding-type identity

Four Holding types — Account, Purse, Strongbox, Vault — each carry a distinct icon and a distinct accent color. The icons are filled-flat in the brand-mark register (no Tabler-style line icons; the tally stick set the visual family). The accent colors are derived from each icon's natural-object material: limestone, auburn leather, iron, brushed steel.

Trust gradient as a brand principle

Iconography is honest about what each Holding type actually is. This is the confirmation-honesty principle (PROCESS.md §3 gauntlet Q4) extended into iconography — the icon doesn't soften reality, it surfaces it.

  • Account — funds at a foreign institution you can't fully trust. The icon is a classical-pediment institutional building: cold, distant, functional. Limestone-grey says "stone" not "wood."
  • Purse — your everyday wallet. Auburn leather drawstring pouch: warm, used, personal. Closest to the user emotionally.
  • Strongbox — your cold-storage hardware wallet. Iron chest with a lock plate: heavy, functional, key in your hand.
  • Vault — your ceremonial multi-key hold. Bank vault door with three chunky handles, six inner locking tubes, and a brass hub: complex, rarely opened, weighty. Hinged on the left (mechanical pivot axis).

The ordering Account → Purse → Strongbox → Vault is also a trust gradient: from "you don't hold the keys" to "you hold the keys ceremonially." The visual temperature confirms it — Account cool-grey and Vault cool-grey bracket the warm pair (Purse auburn, Strongbox iron-warm-grey) in between. The two cool-grey extremes share a brand-cohesion note: brass appears on both the Vault icon (the central hub) and the Purse watch-only cord. Brass marks "this is the channel that holds something valuable." The brass thread also drives --color-holding-vault at the token level: the per-row stripe in the Holdings list and the picker's icon-border for Vault both render brass, extending the thread out from icon-detail to surface-wide identity. Vault icon body stays brushed-steel internally — the brass move is on the framing, not on the icon's body fill.

Usage rule — icons in rich moments, stripes in lists

The icons appear at large sizes only — Add-Holding popup cards (≥64 px), Holding detail headers (≥32 px), pickers and onboarding. In Holdings list rows, the icon is omitted; a 4 px-wide colored stripe on the left edge carries the Holding-type identification at a glance. The text label does the precise identification.

The 16 px constraint was deliberately abandoned. The four Holding colors are not tuned for tiny iconography; they are tuned for stripe legibility and large-icon material authenticity. The brand-mark continues to handle ≤16 px favicon-class iconography via its own size-system rules (see tallykeep_brand_mark_v1_lock §2).

Implication. The v1.1 palette is unconditionally usable as body-text-on-light-surface — no weight bumping required, no surface caveats. The cost was a uniform shift toward deeper saturation; the deep-leaf-bright register reads as "weighted" rather than "muted," which is congruent with the restraint philosophy.

8 · Decisions log

v2 — Holding-type identity (2026-05-10)
Added four Holding-type accent colors (limestone, auburn leather, iron, brushed steel) plus the four canonical SVG icons, locked together as the Holding-type identity system. Iconography style: filled-flat in the brand-mark register. Usage rule: icons at large sizes only (Add-Holding popup, detail headers); list rows use a 4 px colored stripe instead of the icon. The 16 px constraint was deliberately abandoned for Holding icons — colors are tuned for stripe legibility, not tiny iconography. Purse has two SVG variants encoding seed origin: brass cord = external watch-only, dark cord = TallyKeep-managed (on-device keys). Brass appears on both the Vault hub and the Purse watch-only cord as a deliberate brand-cohesion thread.
v1.1 — AA tightening (2026-05-10)
Initial v1 values failed WCAG AA 4.5:1 for body text on page-bg: success #2e8a3f at 4.16:1, warning #cc7a14 at 3.15:1. (Documented ratios in the original v1 lock were inflated; corrected here.) Rather than document caveats around weight-bumping or surface restrictions, all four mains were tuned darker so they comfortably pass AA on every surface the product uses (page-bg, white, cream). Soft / border / text-on-soft tokens unchanged. Rejected previous mains kept on file in §7 for context.
Restraint over decoration
Color earns its appearance. Generic web semantics (Tailwind defaults) treat green/red as decorative; banking-grade restraint reserves them for the moments that map to consequence. Specifically wired to gauntlet Q4 (confirmation honesty) — green requires verifiable-true on-chain depth, never composed/signed/broadcast intermediates.
Hue family — warm-tuned, leaf-bright
Drawn from natural references (holly leaf, holly berry, autumn maple, cornflower). Saturated enough to retain signal strength; not so saturated they look like Tailwind. The v0 muted sketch ran too quiet — danger especially needed brighter chroma to actually stop the eye on destructive actions.
Four-token pattern (main / soft / border / text-on-soft)
Chose four tokens per semantic over the prior two-token (main / soft) pattern. Rationale: real banner / badge components need a separator (border) and a legible text color inside a soft-bg fill (text-on-soft). Three of those four are derivable from the main, but explicit tokens prevent accidental low-contrast pairings.
Info kept provisional
Cornflower works on its own. As a fourth color in a warm-and-earthy palette, it's the hue-outlier. Open question (deferred): collapse info into muted text + brand-color accent, dropping the slot entirely. Held provisional pending a real screen test where a hosted-tier disclosure or attribution surface forces the call.
Rejected — decorative defaults (Tailwind-ish)
#16a34a success, #ca8a04 warning, #dc2626 danger, #0284c7 info. Pasted-on against cream; would have made TallyKeep look like a generic SaaS. Kept on file in tokens.css as the rejected baseline.
Rejected — v0 muted (rust / dusty)
#3a7d44 sage, #b8741c deep amber, #a23a1c rust, #4a6c8a dusty blue. Elegant but undersold danger and confirmation. Worth keeping in mind if the brand voice ever leans more contemplative — could be a future variant for read-only / archive surfaces.
Rejected — v1 leaf-bright (pre-AA fix)
#2e8a3f success, #cc7a14 warning, #c4361c danger, #3672a8 info. Right hue family, right brand feel — but success and warning failed AA body on page-bg. Replaced by the deeper variants above 2026-05-10. The aesthetic gap between v1 and v1.1 is small; the accessibility gap is decisive.

9 · Open items

Info collapse decision
Drop the info slot entirely; ride neutral notices on muted text + brand-color accent. Defer until a real screen — hosted-tier disclosure, "via Mempool · 2m ago" attribution, tooltip — forces the call. Will produce a v2 lock if info is dropped or repurposed.
Stripe-accent reinforcement (proposed)
Idea raised at v2 lock: add a small horizontal accent on each list-row stripe to echo the Holding-type icon. Vault stripe → small brass bar (echoes the brass hub). Purse watch-only → small brass dot (echoes the brass cord). Purse managed → small dark dot (echoes the dark cord). Account / Strongbox → plain stripe, no accent. Adds the brand-thread to the list view without requiring icons. Prototype before next lock.
Dark mode
Light-mode only here. Wordmark §7 has dark-mode rules; mark §8 flags dark mode as untested for the icon. The semantic palette will need dark-mode counterparts (deeper-but-not-saturated softs, lighter mains) before any dark theme ships. Tackle alongside dark mode for the icon and surfaces — not a values-only extension.
Confirmation-state spine
Open question: does Bitcoin confirmation depth (composed → signed → broadcast → confirmed-1 → confirmed-6 → effectively-final) ride on the four-color semantic palette, or does it get its own dedicated 3-token spine (pending / partial / final)? Affects how the Send and Activity flows visualize progression. Best resolved in the Send-iteration design pass.
Real-screen check before final lock
This palette is locked as the working v1. Recommendation: run it through one full mockup (Onboarding + Home populated) and one banner-heavy surface (security-discrepancy banner on home) before promoting to a "final" status. Anything that misbehaves under realistic content density surfaces only there.
Trademark
Carried over from mark/wordmark open lists — palette doesn't change the trademark question, but confirms the brand combination (mark + wordmark + color system) that needs registering pre-public-announcement. Lawyer-question for ship-gate.
