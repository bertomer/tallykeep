# ADR-0021 — Pairing handshake crypto

- **Date:** 2026-05
- **Status:** Accepted
- **Decided by:** Rémy
- **Authored by:** Claude, during unlock-flow-cleanup design session
  2026-05-27
- **Migrated from:** `pre-implementation.md` Open item

## Context

When the mobile client scans a QR generated by the user's TallyKeep
web admin to pair, the QR carries a pairing URL and a single-use
token. Once redeemed, the backend issues a long-lived per-device
credential the phone stores in Keychain/Keystore. The
`pre-implementation.md pairing-handshake-crypto` leading direction
established this shape (WhatsApp Web / Signal device-link pattern);
this ADR closes the open sub-questions:

- Pairing token format
- TTL value, single-use vs N-use, refresh policy
- Server-side rate limit
- Per-device credential format
- Whether an additional desktop-side approval step is worth the
  friction

The threat model is per `01_architecture.md §"Network security
posture"`: the backend binds to `127.0.0.1` (or to the deployment's
reverse proxy / Tailscale / Cloudflare Tunnel for LAN/remote
exposure). The pairing exchange happens over TLS when the operator
runs HTTPS (Umbrel / Tailscale / Caddy / hosted-tier, the
realistic path) or over plain HTTP on a trusted LAN (the dev /
self-host-without-cert path). Adversary model: someone on the same
LAN intercepting the pairing token during its TTL window. With TLS
this is mitigated; with plain HTTP this requires the user's LAN to
be trusted.

The earlier framing considered PAKE-style or Noise-style mutual-
auth handshakes as the "more robust" alternative. They add real
ceremonial cost (mutual-auth round-trips, ephemeral key generation,
replay protection) for a threat the deployment posture already
narrows. The trade-off doesn't pay off in the same-LAN /
HTTPS-when-deployed-properly posture, and the WhatsApp Web pattern
is what users already understand.

The pairing flow is initiated by the **web admin** (per ADR-0020),
not by any other surface.

## Decision

### Pairing token

- **Format.** **12 base32 characters** (RFC 4648, no padding,
  uppercase). Roughly 60 bits of entropy. Displayed grouped
  `XXXX-XXXX-XXXX` for human readability, the hyphens are
  cosmetic, stripped on input. Case-insensitive on the wire (the
  base32 alphabet is uppercase but accepting lowercase on
  redemption is operator-friendly).
- **Why 12 chars and not 256 bits.** The mobile pairing UI
  carries a manual-entry fallback for cases where the QR scan
  fails (camera issues, low light, screen glare, lockscreen
  awkwardness). 256-bit base32 is 52 characters, typeable in
  principle, hostile in practice. 12 characters is the
  banking-app norm, comfortably typeable, and the security model
  doesn't need more: with a 90-second TTL and 10 redemption
  attempts per minute per source IP, brute-forcing 2^60 is on the
  order of 10^14 years. The QR-possession + short-TTL +
  rate-limit composition does the load-bearing work.
- **TTL.** 90 seconds from issuance. Refresh-on-demand button
  always available on the QR display. The QR auto-refreshes every
  60 seconds while the pairing screen is open (so the operator
  doesn't have to think about staleness; an expired token shown
  on screen is the friction case).
- **Single-use.** Server marks the token consumed on first
  redemption; replay attempts return `409 Conflict`.
- **Server-side store.** Token stored hashed (SHA-256 is
  sufficient; the token is high-entropy enough for SHA, no
  slow-KDF needed) in a pairing-attempt table with `issued_at`,
  `consumed_at`, `device_label` set on redemption, IP /
  user-agent captured for the operator's audit view.

### Pairing URL (operator-configured)

The QR payload encodes the URL the phone POSTs to:
`<pairing_external_url>/api/v1/admin/pairing/redeem?t=<token>`.
`<pairing_external_url>` is **not auto-derived** from the
browser's current origin, that path silently breaks for the
common case where the operator runs the install wizard on a
laptop pointed at `http://localhost:8000` while the phone needs
to reach the server over the LAN or via Tailscale.

- **Storage.** `pairing_external_url` is stored in the
  configuration table (alongside `server_label`). Editable from
  the web admin's install wizard (step 3) and from the paired-
  devices view's "Pair a new device" sub-route.
- **Default.** Pre-filled from `window.location.origin` when the
  operator opens step 3. Operator-editable; the wizard does not
  advance until the operator confirms.
- **Warnings.** The wizard surfaces a yellow warning if the URL
  is any of:
    - `localhost`, `127.0.0.1`, or `0.0.0.0` (phone cannot reach
      loopback)
    - a `.local` mDNS name (iOS reliable, Android flaky, never
      cellular)
    - a private-RFC1918 address (`10.*`, `192.168.*`, `172.16-31.*`)
      without HTTPS (phone-side TLS pinning may reject plain HTTP
      to private IPs)
  The warning is informational, not blocking, the operator may
  legitimately want LAN-only pairing on a trusted Wi-Fi.
- **Examples shown inline:**
    - Umbrel MagicDNS: `https://umbrel.local` (iOS) or the
      Umbrel-issued Tailscale hostname (Android-friendly)
    - Tailscale MagicDNS: `https://tallykeep.<tailnet>.ts.net`
    - Cloudflare Tunnel: `https://tallykeep.<your-domain>.com`
    - LAN with self-signed: `https://192.168.1.42:8000` (phone
      will warn on cert)
    - LAN plain HTTP (dev only): `http://192.168.1.42:8000`

### Pairing redemption

The phone POSTs the token to the pairing endpoint with metadata
the operator will need to recognize the device in the paired-
devices list:

- `device_label` (free-form, default "Phone" or platform-derived
  "iPhone" / "Pixel 7", the user can edit later)
- `platform` (`ios` | `android`)
- `app_version`

Backend validates the token (existence, non-consumed, within TTL),
marks it consumed, issues the per-device credential, returns it
in the response. The phone stores the credential in
Keychain/Keystore.

### Per-device credential

- **Format.** Opaque bearer, 256-bit cryptographically random,
  base64url-encoded. Server-side validation against a
  `paired_device` table keyed by `device_id` (UUID), with
  `credential_hash` (SHA-256), `issued_at`, `last_seen_at`,
  `device_label`, `platform`, `revoked_at`. The credential never
  appears in any log or response after the pairing POST returns.
- **Why opaque, not JWT.** JWT's stateless-validation advantage
  doesn't pay off here. Revocation already requires a DB lookup
  (the `revoked_at` check), so the validation path is going to
  the database either way. Opaque bearer is the simpler primitive
  to reason about, harder to mis-issue, and trivial to rotate
  (delete the row, the credential is dead, no JWT-blacklist
  dance).
- **Rotation.** Out of scope for iteration v1. Per-device
  credentials are revoked from the web admin's paired-devices
  view; the phone re-pairs when revoked. Active rotation (issue
  new credential, retire old without re-pair) lands when needed.

### Rate limit

Server-side, per-IP, on the pairing-redemption endpoint:

- 10 pairing-redemption attempts per minute per source IP.
- 100 per hour per source IP.
- After lockout: 5-minute cooldown, then attempts resume at half
  rate until the next clean hour.

Plus a global per-server-instance ceiling so a misconfigured
device storming the endpoint can't degrade the rest of the
service. Specific numbers sharpen at implementation; the
principle is "make brute-force of the token completely infeasible
in the 90-second window while not interfering with a user
re-trying after a fat-finger".

### No desktop-side approval step

After the phone successfully redeems the token, the web admin
shows "Phone paired" and proceeds to the paired-devices view. No
additional "approve this pairing from phone XYZ?" confirmation
on the web admin side. The security argument is:

- The QR was generated by the web admin, which the operator just
  authenticated to.
- The QR is single-use and short-lived.
- Same-LAN or HTTPS constraint narrows the window further.

Adding an approval step adds ceremony without changing the threat
model, anyone who scanned the QR would presumably also approve
their own scan from the same session. UBS / WhatsApp / Signal
don't add this step and the security delta is zero.

## Replaces the existing `/api/v1/pairing/issue` endpoint

A pre-iteration backend ships an unauthenticated
`POST /api/v1/pairing/issue` returning a 43-character base64url
token (~256 bits, mixed case, e.g.
`ph5JZA8DD7lcN5UTyD-Ucfi8TgEeKRpuC2qhXMmom1A`). The operator's
current daily flow is `curl + paste into mobile`, which this
iteration retires.

Nothing has shipped to the public; there is no retro-compat
requirement to maintain. The clean cutover is:

- **Delete `POST /api/v1/pairing/issue`** in this iteration's
  backend work. Same for its legacy redeem counterpart. No
  deprecation period, no coexistence dance.
- **Mobile's pairing-key input updates to the new short format
  in lockstep**, as part of this iteration (small mobile-side
  scope bullet in `next_iteration.md`): accept 12-char base32
  with optional hyphens, POST to the new
  `POST /api/v1/admin/pairing/redeem` endpoint, store the
  per-device credential as before.
- The admin web UI, the backend, and the mobile pairing-key
  field all ship the new format together. After this iteration
  closes, the only way to pair is the new flow; the curl path
  is gone.

## Consequences

- **Backend endpoints land in iteration v1:**
  - `GET /api/v1/admin/pairing/url`, authenticated. Returns
    `{ pairing_external_url: "..." }` or `{ pairing_external_url:
    null }` if not yet set. Used by the wizard to pre-fill.
  - `PUT /api/v1/admin/pairing/url`, authenticated. Accepts
    `{ pairing_external_url: "..." }`. Server validates URL
    syntax + returns the warning flags listed above
    (`localhost`, `.local`, `private_ip_no_tls`) so the wizard
    can render them; does not block on warnings.
  - `POST /api/v1/admin/pairing/qr`, authenticated by session
    token. Uses the stored `pairing_external_url`; if unset,
    returns `409 Conflict` with `{ error: "pairing_url_unset" }`.
    Returns `{ token, qr_payload, key_grouped, expires_at }`,
    `qr_payload` is the URL the QR encodes
    (`<pairing_external_url>/api/v1/admin/pairing/redeem?t=<token>`),
    `key_grouped` is the bare 12-char token formatted
    `XXXX-XXXX-XXXX` for the manual-entry display.
  - `POST /api/v1/admin/pairing/redeem`, unauthenticated (token
    is the bearer of authority during its TTL). Accepts the
    token (with or without hyphens, case-insensitive) plus
    device metadata. Returns the per-device credential.
  - `GET /api/v1/admin/devices`, authenticated. Returns paired
    devices with metadata.
  - `DELETE /api/v1/admin/devices/{device_id}`, authenticated.
    Revokes the credential.
- **Legacy `POST /api/v1/pairing/issue` and its redeem
  counterpart are deleted in this iteration** (per the Replaces
  section above). After Tier 1 ships, the only pairing path is
  the new admin flow.
- **Mobile pairing-key input updates as a small in-scope
  bullet** rather than deferring to the mobile-side iteration.
  Keeps the cutover atomic; no broken intermediate state.
- **`pre-implementation.md pairing-handshake-crypto` is closed.**
  Slug migrates here per the convention.
- **No PAKE / Noise / ephemeral-key handshake.** Single-use bearer
  token over TLS (or trusted-LAN HTTP) is the model. Documented
  here so a future agent doesn't propose the "more robust"
  alternative without first reading the threat-model reasoning.
- **Revisit trigger.** Worth revisiting when remote pairing lands
  (`backlog/remote-access-for-self-hosters.md`) and the
  same-LAN / HTTPS-by-default deployment assumption no longer
  holds. The handshake may need to harden at that point.
